# 注册「该邮箱已注册」且云库无记录 — 根因与修复说明

## 一、根因结论（已确认）

- **「该邮箱已注册」来自前端本地逻辑，不是 uni-id 或云数据库返回。**
- **注册流程此前没有调用任何云函数/云对象，也没有向云数据库写入。**

### 具体原因

1. **提示来源**  
   在 `components/sheets/LoginRegisterSheet.vue` 中，注册时用 **本地 storage 的 `users` 数组** 判断邮箱是否已存在：
   - `getStoredUsers()` 读取 `uni.getStorageSync('users')`（本地 JSON 数组）
   - 若数组中已有相同 `email`，则直接显示「该邮箱已注册」并 `return`，**不会请求云端**。

2. **数据落库**  
   原注册逻辑仅执行：
   - `users.push(newUser)`  
   - `uni.setStorageSync('users', JSON.stringify(users))`  
   即只写入**本设备 storage**，没有调用云函数/云对象，因此**云数据库 uni-id-users 表不会有新记录**。

3. **为何会看到「该邮箱已注册」**  
   同一设备上曾用该邮箱做过一次「注册」（当时只是写入了本地 `users`）。再次用同一邮箱注册时，前端在本地数组中查到该 email，于是提示「该邮箱已注册」，与云数据库是否真有该邮箱无关。

## 二、已做修复（概要）

### 1. 云对象 `register`（`uniCloud-aliyun/cloudfunctions/register/index.obj.js`）

- **`registerUser(email, password, nickname)`**
  - 在云数据库 **uni-id-users 表** 中按 `email` 查询是否已存在。
  - 若已存在：返回 `{ errCode: 'EMAIL_EXISTS', errMsg: '该邮箱已注册' }`（**此时才是真正的「云端」邮箱已注册**）。
  - 若不存在：将新用户写入 **uni-id-users 表**（密码经 pbkdf2 加密），并可选返回 `newToken`、`userInfo`。
- **`login(email, password)`**
  - 在 uni-id-users 表中按邮箱+密码校验，成功则返回 `newToken` 与 `userInfo`，便于与 uni-id token 体系一致（如 `set_enable_sending` 等依赖 token 的接口可继续使用）。

### 2. 前端 `LoginRegisterSheet.vue`

- **注册**：改为调用 `uniCloud.importObject('register').registerUser(...)`，根据返回的 `errCode` 显示「该邮箱已注册」或成功并关闭。
- **登录**：改为调用 `uniCloud.importObject('register').login(...)`，根据 `errCode` 显示「该邮箱尚未注册」「密码不正确」或成功。
- 成功时：将 `newToken` 写入 `uni_id_token` / `uni_id_token_expired`，并 `mutations.setUser(userInfo)`（含 `_id`），与现有 store 和云端接口兼容。

这样，「该邮箱已注册」只会在**云端 uni-id-users 表里确实已有该邮箱**时出现，且新用户会真实写入云数据库。

## 三、你可做的排查与验证

### 1. 确认云库中是否已有该邮箱（含禁用账号）

- 在 **uniCloud 控制台 → 云数据库 → uni-id-users 表**：
  - 按 `email` 字段查询你用来注册的邮箱。
  - 若有记录（包括 `status: 1` 禁用），则云端会正确返回「该邮箱已注册」，且不会再次插入。

### 2. 查看云函数/云对象日志

- **uniCloud 控制台 → 云函数/云对象 → register → 日志**：
  - 若之前从未接入云端注册，旧版不会产生调用记录。
  - 修复后：每次点击注册应有一条 `register` 调用；若写入失败会有报错栈，便于区分「邮箱已存在」与「权限/校验/网络」等问题。

### 3. Schema 与权限（当前配置是否合理）

- **表级 permission**  
  - `create: false` 表示**禁止客户端通过 clientDB 直接 create**。  
  - 注册是由**云对象**里 `db.collection('uni-id-users').add()` 完成的，走的是**服务端 MongoDB API**，**不受 schema 的 create 限制**，因此无需改 schema 的 create。
- **验证规则**  
  - `email` 的 `format: "email"`、`password` 的 `bsonType: "password"` 等主要约束的是 clientDB/JQL；云对象里用 `add()` 写入时仍会做格式校验，但不会因为「create 权限」而失败。

### 4. 密码加密与生产环境

- 当前云对象使用环境变量 `UNI_ID_PASSWORD_SECRET`（若未配置则用代码中的默认密钥）。
- **生产环境**：请在 **云空间 → 云函数/云对象 → register → 配置 → 环境变量** 中设置 `UNI_ID_PASSWORD_SECRET`，避免使用默认值。

### 5. 本地旧数据对提示的影响

- 若希望「该邮箱已注册」只代表云端状态，可清理本机 storage 中的旧 `users`，避免和旧逻辑混淆：
  - 开发者工具 / 运行到真机前，在应用内或调试里清除 storage，或删除 key `users`。
- 修复后的逻辑已不再读取本地 `users` 做注册判断，仅以云端返回为准。

## 四、小结

| 问题 | 结论 |
|------|------|
| 「该邮箱已注册」是谁返回的？ | **原先是前端根据本地 storage 的 `users` 数组判断的；** 修复后，该提示只会在**云端 uni-id-users 表已存在该邮箱**时由云对象返回。 |
| 为什么云库没有新用户？ | 原先注册**只写本地 storage**，未调用云端，所以云库不会有新记录。 |
| 现在注册会写云库吗？ | 会。注册改为调用云对象 `register.registerUser`，新用户会写入 **uni-id-users 表**。 |
| 需要改 schema 的 create 吗？ | 不需要。注册在云对象内用 MongoDB API 写入，不受 schema 的 `create: false` 限制。 |

按当前修改部署并重新测试注册流程后，再在云数据库 uni-id-users 表中即可看到新用户记录；若仍提示「该邮箱已注册」，则可在该表中查到对应邮箱的已有文档。
